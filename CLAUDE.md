# Проект: База правовой аналитики

## Назначение
База знаний по праву РФ: нормы (НПА), судебная практика, правовые заключения.
Возможности: полнотекстовый и фасетный поиск, связи норма↔практика↔заключение,
конструктор заключения с выгрузкой, дашборд по отраслям.
Учебный проект (Цифровая кафедра ДГТУ, Тема №5).

## Стек
- Django 5.x, Python 3.12, PostgreSQL (обязателен для FTS)
- Полнотекстовый поиск: django.contrib.postgres.search, конфигурация 'russian', GIN-индекс
- django-taggit (теги), django-filter (фасеты)
- docxtpl — экспорт заключения в .docx
- Chart.js — дашборд; vis-network — граф связей
- Bootstrap 5 + HTMX, Docker Compose, WhiteNoise

## Соглашения по коду
- **Все идентификаторы (переменные, функции, классы, поля) — только английские**
- Русский язык только в: verbose_name, label, тексте шаблонов, данных
- Каждое поле модели обязательно имеет verbose_name на русском
- В UI пользователь всегда видит human-readable метки, никогда не сырые имена полей
- Тесты обязательны для поиска и конструктора заключения (pytest)
- Контент базы — в data/ (JSON) + management-команда load_legal_data

## Структура приложений
- apps/catalog/    — LegalBranch (отрасли права)
- apps/knowledge/  — Norm, CourtCase, LegalOpinion + M2M связи
- apps/search/     — FTS поиск + фасеты (django-filter)
- apps/builder/    — конструктор заключения, экспорт docx
- apps/analytics/  — дашборд Chart.js
- apps/accounts/   — аутентификация, 3 роли (Groups)

## Роли
- "Редактор базы" — добавляет/правит нормы, практику, теги
- "Аналитик" — ищет, строит заключения, видит дашборд
- "Читатель" — только поиск и просмотр

## Команды
- Запуск: `docker compose up`
- Тесты: `pytest`
- Загрузка контента: `python manage.py load_legal_data`
- Суперпользователь: задаётся в .env (DJANGO_SUPERUSER_*)

## Чего НЕ делать
- Не строить нейросетевой поиск/ИИ-резюме
- Не оставлять базу пустой — минимум 40–60 наполненных записей
- Не хардкодить отрасли/теги — брать из справочников
- Не писать идентификаторы на русском языке
