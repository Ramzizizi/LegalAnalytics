# План разработки проекта с помощью Claude Code
### Конструктор типовых юридических документов (Django + docxtpl) — команда 5 человек

---

## 0. Как устроен этот план

Claude Code — это агент, который пишет код прямо в вашем репозитории: читает файлы, вносит правки, запускает команды и тесты, но **спрашивает разрешение перед каждым изменением**. Поэтому ключ к скорости — не «писать код руками», а грамотно ставить задачи и держать контекст проекта в одном файле (`CLAUDE.md`).

План построен так: сначала настраиваем окружение и каркас, затем идём по модулям. Для каждого модуля дан **готовый промпт**, который можно почти дословно отдать Claude Code. Работаем итерациями: один модуль доведён до рабочего состояния → коммит → следующий.

> Главный принцип: **сначала один сквозной сценарий end-to-end (форма → .docx → .pdf), потом расширение.** Не просите Claude Code построить всё сразу — это даёт хрупкий результат.

## 1. Подготовка окружения (день 1)

### 1.1. Что нужно установить (на машине разработчика — игровой ПК на Windows подойдёт)

| Инструмент | Зачем | Команда / источник |
|---|---|---|
| **Claude Code** | Сам агент | см. ниже |
| **Git** | Версионирование, командная работа | git-scm.com |
| **Python 3.12+** | Рантайм проекта | python.org |
| **Docker Desktop** | PostgreSQL + LibreOffice в контейнере, демо на защите | docker.com |
| **VS Code** (опц.) | Редактор + терминал для Claude Code | code.visualstudio.com |
| **LibreOffice** | Конвертация .docx → .pdf | только внутри Docker-образа |

### 1.2. Установка Claude Code

Нужен аккаунт **Claude.ai (Pro или Max)** либо Claude Console — на бесплатном плане Claude Code недоступен.

- **Нативная установка (рекомендуется, Node.js не требуется):**
  - Windows (PowerShell): `irm https://claude.ai/install.ps1 | iex`
  - macOS/Linux/WSL: `curl -fsSL https://claude.ai/install.sh | bash`
  - macOS/Linux через Homebrew: `brew install --cask claude-code`
- **Через npm (нужен Node.js 18+):** `npm install -g @anthropic-ai/claude-code`

Проверка и вход:
```bash
claude doctor      # диагностика установки
claude             # запуск; при первом старте попросит /login
```

> На рабочем Mac с MDM ставить не нужно — вся разработка идёт на личном ПК. Для защиты проект поднимается через Docker на любом ноутбуке.

### 1.3. Первый запуск в проекте

```bash
mkdir legal-docs-constructor && cd legal-docs-constructor
git init
claude
```
Внутри сессии первым делом:
```
> /init
```
Команда `/init` создаёт файл `CLAUDE.md` — «постоянную память» проекта, которую Claude Code читает каждый раз. Дальше мы её наполним (раздел 3).

**Полезные команды внутри сессии:** `/help`, `/clear` (очистить контекст между задачами — делайте часто), `/status` (лимиты), `Shift+Tab` (переключение в plan mode — Claude сначала предлагает план, потом пишет код).

## 3. CLAUDE.md — главный файл для агентной разработки

Это самая важная настройка. Хороший `CLAUDE.md` экономит десятки повторных объяснений: контекст лежит в файле, а не в каждом промпте. Попросите Claude Code заполнить его по этому образцу:

```markdown
# Проект: Конструктор типовых юридических документов

## Назначение
Веб-приложение: юрист выбирает шаблон → заполняет форму →
система подставляет данные → выгружает готовый .docx и .pdf.
Учебный проект (Цифровая кафедра ДГТУ). Право РФ: ГК, ГПК, АПК.

## Стек
- Django 5.x, Python 3.12
- docxtpl + Jinja2 — генерация .docx из Word-шаблонов
- LibreOffice headless — конвертация .docx → .pdf
- PostgreSQL (prod/демо), SQLite (локально)
- Bootstrap 5 + HTMX, django-htmx
- Docker Compose

## Соглашения
- Код и комментарии — по-русски там, где это доменные термины.
- Все строки/документы — UTF-8.
- Валидаторы реквизитов (ИНН, даты, суммы) — в apps/*/validators.py.
- Тесты обязательны для движка генерации и валидаторов (pytest).
- Никаких секретов в коде — только через .env.

## Команды
- Запуск: `docker compose up`
- Тесты: `pytest`
- Миграции: `python manage.py makemigrations && migrate`

## Чего НЕ делать
- Не добавлять оплату, чат, ИИ-генерацию текста (вне объёма).
- Не хардкодить пути к шаблонам — брать из БД.
- Не коммитить сгенерированные документы и .env.
```

> Совет: держите `CLAUDE.md` коротким и точным. Когда меняются соглашения — обновляйте файл, а не повторяйте правила в промптах.

## 4. Командная работа над одним кодом (5 человек)

Агент пишет быстро, поэтому узкое место — **слияние изменений**. Дисциплина:

1. **Ветки по модулям.** Каждый работает в своей ветке (`feature/templates-engine`, `feature/documents-form` и т.д.). Claude Code сам делает коммиты по просьбе («закоммить изменения с понятным сообщением»).
2. **Commit перед каждой сессией.** Это «страховка»: всегда можно откатиться, если агент сделал не то.
3. **Pull request + ревью.** Любой код от Claude Code просматривает второй человек перед слиянием в `main`. Агент ускоряет написание, но ответственность за качество — на команде.
4. **Параллельность через git worktree** (по желанию): несколько веток одновременно в разных папках, чтобы агенты не мешали друг другу.
5. **Общий `CLAUDE.md`** в `main` — единый источник правды для всех.

**Кто что ведёт (по ролям из методички):**

| Участник | Ветки/модули | Артефакты защиты |
|---|---|---|
| Аналитик / тимлид | требования, BPMN, ER-модель, фикстуры | разделы 1–2 записки, ROI |
| Архитектор / вед. разработчик | каркас, Docker, `templates_engine` (ядро) | раздел 3.1–3.2 |
| Разработчик (UI) | `documents` (формы, HTMX), `catalog` | раздел 3.3 |
| Тестировщик / QA | `tests`, чек-лист, баг-репорты | раздел 3.5, видео |
| Тех. писатель / докладчик | README, скриншоты, презентация, доклад | оформление, защита |

## 5. Пошаговый план реализации (готовые промпты для Claude Code)

Каждый этап: отдайте промпт → проверьте результат → попросите тесты → коммит → `/clear`.

### Этап 1. Каркас проекта и Docker (неделя 2)
```
Создай каркас Django 5 проекта по структуре из CLAUDE.md.
Настрой config/settings с переключением SQLite/PostgreSQL через
переменные окружения (.env). Добавь приложения catalog,
templates_engine, documents, accounts. Подготовь Dockerfile и
docker-compose.yml с сервисами web (Django+gunicorn) и db (Postgres),
с томом для данных. В образ web доустанови libreoffice для конвертации
в PDF. Добавь .gitignore и .env.example. Проверь, что `docker compose up`
поднимает приложение.
```

### Этап 2. Модели данных и админка (неделя 2)
```
Спроектируй и создай модели:
- catalog: Контрагент (наименование, ИНН, КПП, адрес, контакты, тип
  физ/юр), Юрист.
- templates_engine: ШаблонДокумента (название, тип, файл .docx,
  описание), Переменная (шаблон, ключ, подпись, тип данных,
  обязательность).
- documents: СгенерированныйДокумент (шаблон, автор, дата, контрагент,
  значения JSON, файлы docx/pdf, статус).
Сделай миграции, зарегистрируй всё в Django admin с удобным
отображением и поиском. Напиши тесты на создание объектов.
```

### Этап 3. Ядро генерации end-to-end (неделя 3) — КЛЮЧЕВОЙ ЭТАП
```
Реализуй движок генерации в apps/templates_engine/services.py:
1) функция render_docx(template_path, context) на docxtpl — подставляет
   значения в .docx-шаблон и возвращает готовый файл;
2) функция convert_to_pdf(docx_path) — конвертирует через headless
   LibreOffice (subprocess), возвращает путь к PDF.
Добавь один тестовый Word-шаблон "Договор оказания услуг" с тегами
{{ contragent.name }}, {{ summa }}, {{ data }} и т.п. в word_templates/.
Напиши management-команду или тест, который генерирует документ из
тестовых данных и проверяет, что .docx и .pdf создаются. Покажи рабочий
результат от формы данных до файла.
```
> Контрольная точка проекта: если этот этап работает — главный риск снят.

### Этап 4. Форма ввода и генерация из UI (неделя 4)
```
Сделай страницу: выбор шаблона → динамическая форма по его Переменным →
кнопка "Сгенерировать". После отправки система рендерит .docx и .pdf,
сохраняет СгенерированныйДокумент и даёт ссылки на скачивание.
Используй Django-формы, Bootstrap 5 и HTMX (без отдельного React):
форма и обновление списка — без перезагрузки страницы. Не используй
HTML <form> внутри SPA-логики там, где есть HTMX-обработчики; следуй
серверному рендерингу Django.
```

### Этап 5. Валидация реквизитов (неделя 4)
```
Добавь валидаторы в apps/catalog/validators.py:
- ИНН: 10 цифр (юрлицо) с проверкой контрольной цифры по коэффициентам
  [2,4,10,3,5,9,4,6,8]; 12 цифр (физлицо/ИП) с двумя контрольными
  цифрами. Невалидный ИНН — понятная ошибка в форме.
- дата не в прошлом там, где это требуется; сумма > 0.
Подключи валидаторы к формам и моделям. Напиши параметризованные
pytest-тесты с валидными и невалидными значениями ИНН.
```

### Этап 6. Роли и доступ (неделя 4–5)
```
Настрой роли через Django Groups/Permissions:
- "Администратор шаблонов" — управляет шаблонами и переменными;
- "Юрист" — создаёт документы, видит свои и общие;
- "Помощник" — только заполняет и просматривает.
Закрой вьюхи декораторами login_required и permission_required.
Реализуй фильтрацию: пользователь видит свои сгенерированные документы.
Добавь тесты на разграничение доступа.
```

### Этап 7. Журнал документов (неделя 5)
```
Сделай страницу "Журнал" со списком сгенерированных документов:
фильтры по типу, контрагенту, дате, автору; живой поиск через HTMX;
повторное скачивание .docx/.pdf. Добавь пагинацию.
```

### Этап 8. Остальные шаблоны (неделя 5)
```
Добавь ещё 3–4 Word-шаблона: договор поставки, претензия, доверенность,
исковое заявление. Для каждого заведи Переменные. Проверь генерацию
по каждому типу. (Содержание шаблонов согласуем с преподавателем-юристом
отдельно — твоя задача техническая: корректная подстановка.)
```

### Этап 9. Демо-данные и стабилизация (неделя 6)
```
Создай fixtures с демо-данными (контрагенты, юристы, шаблоны,
несколько готовых документов), чтобы на защите база была наполнена.
Добавь команду загрузки фикстур. Проверь полный сценарий на чистой
базе через docker compose.
```

## 9. Типичные грабли (и как их избежать)

- **Слишком много в одном промпте.** Дробите на модули; после каждого — `/clear`, чтобы не «засорять» контекст.
- **Контекст в промптах вместо CLAUDE.md.** Повторяющиеся правила — в `CLAUDE.md`, иначе тратите время на объяснения каждый раз.
- **Слияние без ревью.** Код от агента всегда читает второй человек перед merge в `main`.
- **Нет коммита перед сессией.** Коммитьте до начала задачи — это кнопка «отменить».
- **PDF «съезжает».** Встройте нужные шрифты (Times New Roman / PT Astra) в Docker-образ; проверьте на машине защиты.
- **Демо зависит от интернета.** Показывайте локально через Docker; облако — максимум как дополнение.
- **Юридическая точность.** Claude Code отвечает за подстановку, а не за «правильность» формулировок — содержание шаблонов согласуйте с преподавателем-юристом.
